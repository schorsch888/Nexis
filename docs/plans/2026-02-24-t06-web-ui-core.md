# Phase 4 P1: T06 Web UI 核心实现计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在 `apps/web` 现有 React + TS + Vite 骨架上，交付可用的核心 Web UI（登录、租户切换、房间列表、消息流、成员管理、搜索）并打通 REST + WebSocket 基础集成。

**Architecture:** 采用“壳层布局 + 页面路由 + 领域状态 + API/WS 客户端”四层架构。登录阶段先基于当前后端可用能力（Bearer JWT）实现可运行会话流，再通过可替换的 `authStore`/`httpClient`/`wsClient` 预留 T07 深化（刷新令牌、权限策略、SSO）扩展点。UI 组件先实现轻量设计系统（tokens + primitives + 业务组件），确保信息架构稳定后再做视觉优化。

**Tech Stack:** React 18, TypeScript, Vite, React Router v6, Zustand, Axios, WebSocket API, Vitest（+ React Testing Library 作为测试补充）。

## 1. 现状分析（As-Is）

### 1.1 前端现状
- 工程为 Vite + React + TS，当前仅有：
  - `App.tsx` 两条路由：`/` 与 `/rooms/:roomId`
  - `Layout` 顶栏 + 简单内容容器
  - `HomePage` 与 `RoomPage` 原型实现（局部 `fetch` 调用）
- 样式层仅 `src/index.css` 全局样式，缺少组件化设计系统。
- 状态管理库 `zustand` 已安装但未使用。
- API 调用混用 `fetch`，缺统一错误处理、鉴权注入、请求取消与重试策略。

### 1.2 后端能力边界（T06 相关）
- 可用 REST：
  - `GET/POST /v1/rooms`
  - `GET/DELETE /v1/rooms/:id`
  - `POST /v1/rooms/:id/invite`
  - `POST /v1/messages`
  - `GET/POST /v1/search`
- 可用 WebSocket：`GET /ws`（当前为 echo 型基础能力，后续可升级事件推送）。
- 鉴权：受保护接口需要 `Authorization: Bearer <token>`。
- 缺口：无独立 `login` 端点、无 `refresh token` 端点（T06 需采用“令牌输入/注入”的过渡登录流）。

### 1.3 关键约束
- 当前目标是“核心可用面”而非完整企业 UX。
- 必须优先保证 IA（信息架构）与 API 契约稳定，避免后续 T07/T08 大幅返工。
- 租户上下文需在 UI 层先落地为“可切换会话上下文（tenantId）”，并通过 header/body 传递（视接口能力选择）。

## 2. 设计方案收敛（2-3 方案）

### 方案 A（推荐）：单体 SPA + 模块化分层（Route + Store + Client）
- 说明：在单一 Vite SPA 内建立 `app/`、`features/`、`shared/` 分层，使用 Zustand 管理 auth/tenant/rooms/messages/search 状态。
- 优点：改造成本低，兼容现有代码，交付速度快，适合 T06 范围。
- 风险：长期复杂度上涨时需再拆分（可在 T08 后演进）。

### 方案 B：引入 React Query + Feature-first
- 说明：新增 `@tanstack/react-query` 作为服务端状态层。
- 优点：缓存/失效/重试模型完善。
- 风险：当前项目尚未使用，T06 引入会抬高迁移和学习成本。

### 方案 C：BFF 驱动（前端仅渲染）
- 说明：新增前端中间层服务，前端只调用统一聚合接口。
- 优点：前端简化，安全边界更清晰。
- 风险：超出 T06 范围，且当前仓库无对应 BFF 基础。

**结论：采用方案 A。** 在不额外引入重型依赖的前提下，先完成核心流程，再在 T07/T08 视复杂度升级数据层。

## 3. 目标信息架构（IA）与路由

### 3.1 路由结构
- `/login`：登录页（token + memberId + tenantId）
- `/app`：应用壳（需登录）
- `/app/rooms`：房间列表
- `/app/rooms/:roomId`：消息流
- `/app/members`：成员管理
- `/app/search`：搜索页
- `/`：重定向到 `/app/rooms` 或 `/login`

### 3.2 应用壳布局
- 顶部：品牌、租户切换、当前用户菜单（登出）
- 左侧：房间列表 + 快速搜索入口
- 主区：路由内容（消息流/成员/搜索）
- 移动端：左侧栏折叠为抽屉

### 3.3 核心状态域
- `authStore`：token/member/session 状态
- `tenantStore`：tenantId、可选租户列表、切换动作
- `roomsStore`：房间列表、分页、当前房间
- `messagesStore`：当前房间消息、发送态、WS 状态
- `searchStore`：query、结果、过滤条件

## 5. API 与 WebSocket 集成设计

### 5.1 认证流程（T06 版）
- 登录页输入：`token` + `memberId` + 可选 `tenantId`
- 登录成功后：
  - 存储到 `authStore`（可选持久化 `localStorage`）
  - 统一 Axios 拦截器注入 `Authorization`
  - 若有 `tenantId`，附加 `X-Tenant-ID`（或按后端约定字段）
- 登出：清理 store + 跳转 `/login`

### 5.2 REST 调用规范
- 建立 `src/shared/api/httpClient.ts`（axios instance）
- 建立 `src/shared/api/endpoints/*`（rooms/messages/members/search）
- 统一错误映射：`4xx` 用户可见错误；`5xx` 通用失败提示
- 请求契约：优先对齐 `docs/en/api/reference.md` 与网关实际返回

### 5.3 WebSocket 连接策略
- `wsClient` 提供 connect/disconnect/reconnect/send
- 首版用途：
  - 连接状态可视化
  - 收到事件后触发消息刷新（当前后端若仍为 echo，保持兼容）
- 重连：指数退避（上限 N 次）+ 手动重试入口

## 8. 风险与缓解

- 风险 1：无正式登录接口导致“登录页”语义偏弱。
  - 缓解：T06 明确为“token-based bootstrap login”，T07 切换到真实 auth API。
- 风险 2：WebSocket 事件协议未完全稳定。
  - 缓解：ws 作为增强层，保留 REST 刷新兜底。
- 风险 3：多租户字段在接口中存在命名差异（`tenant_id` vs `tenantId`）。
  - 缓解：在 API 适配层统一映射，页面不直接依赖后端字段细节。
