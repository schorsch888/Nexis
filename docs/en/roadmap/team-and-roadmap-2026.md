## 1. 顶尖团队结构（人类专家团队）

## 1.1 组织模型（推荐）
推荐采用“核心专家 + 领域小队（Squad）+ 明确 DRI”的矩阵模式。

| 层级 | 角色/团队 | 人数建议 | 核心职责 | 关键交付 |
|---|---:|---:|---|---|
| 平台决策层 | 首席架构师（Chief Architect） | 1 | 架构北极星、技术选型、ADR 审核 | 架构蓝图、关键 ADR、跨队技术对齐 |
| 核心专家层 | 架构师、后端、AI、前端、安全、DevOps、产品专家 | 7 | 各领域技术把关与方案决策 | 领域方案、风险清单、验收门禁 |
| 交付层 | Platform / Realtime / AI Systems / Security / Product 各 Squad | 16-22 | 端到端研发、测试、上线、运维 | 版本交付、SLO 达标、线上稳定 |

## 1.2 核心角色分工与 DRI

| 角色 | DRI 职责 | 周期性输出 | 验收责任 |
|---|---|---|---|
| 架构师 | 总体架构、模块边界、依赖解耦 | 每 Sprint 架构评审、ADR | 核心设计可落地且无阻塞 |
| 后端专家 | 高并发、分布式一致性、数据模型 | 性能报告、瓶颈修复清单 | P95/吞吐/错误率达标 |
| AI 专家 | Agent 编排、A2A、LLM 接入 | Agent 协作策略、模型路由策略 | 任务闭环率、A2A 成功率达标 |
| 前端专家 | Web/Mobile 体验与可用性 | UX 方案、核心页面实现 | 关键流程可用、性能与可访问性达标 |
| 安全专家 | 威胁建模、策略、审计 | 安全基线、审计报告、漏洞清单 | 高危漏洞为 0、审计覆盖达标 |
| DevOps 专家 | 云原生、CI/CD、可观测 | Helm/流水线、SRE 报告 | 发布成功率、MTTR 达标 |
| 产品专家 | 需求拆解、优先级、验收定义 | PRD、里程碑评审、验收标准 | 价值闭环与用户目标达成 |

## 1.3 资源容量（建议基线）
- 总研发容量：18 人
- 每 Sprint（2 周）有效人周：`18 * 2 * 0.75 = 27 人周`（按 25% 非研发开销折算）
- 固定容量预留：
  - 20% 技术债与稳定性
  - 10% 安全与合规

## 2. AI Agent 团队（执行与质量体系）

## 2.1 Agent 编组

| Agent | 定位 | 主职责 | 输入 | 输出 |
|---|---|---|---|---|
| workspace-coder | 实施 Agent | 功能实现、重构、脚手架 | 任务卡、技术方案 | 合并请求、实现说明 |
| architect | 架构 Agent | 技术方案评估、边界定义 | PRD、现有架构 | ADR、模块设计 |
| reviewer | 质量 Agent | 代码审查、回归风险检查 | 代码变更、测试结果 | 审查意见、风险评级 |
| tester | 测试 Agent | 测试设计、回归与性能验证 | 需求、接口、代码 | 测试报告、覆盖率数据 |
| security-guard | 安全 Agent | SAST/依赖漏洞/策略扫描 | 代码、依赖清单、配置 | 漏洞报告、修复建议 |
| doc-writer | 文档 Agent | 文档、Runbook、对外说明 | 方案与实现细节 | 设计文档、运维文档、变更日志 |

## 2.2 人机协同规则
- 每个 P0 模块必须经过：`architect -> workspace-coder -> tester -> reviewer -> security-guard -> doc-writer`。
- 产物门禁：
  - 无测试报告不得进入评审。
  - 有高危安全问题不得发布。
  - 无文档更新不得关闭任务。

## 7. 执行治理与验收机制

## 7.1 周期机制
- 每周：模块例会（进度、风险、阻塞）
- 每 Sprint：计划会、评审会、复盘会
- 每 Phase：里程碑 Gate Review（不过 Gate 不进入下一阶段）

## 7.2 Gate 验收清单（统一）
- 功能：PRD 必要场景全部通过
- 质量：自动化测试通过，阻断缺陷为 0
- 性能：达到模块定义的 P95/吞吐目标
- 安全：高危漏洞为 0，关键审计链路完整
- 文档：设计、运维、回滚预案齐备

## 7.3 责任分配（RACI 简版）
- Responsible：模块 DRI
- Accountable：首席架构师 + 产品专家
- Consulted：安全专家、DevOps、reviewer/tester
- Informed：全体 Squad

## 8. 三种组织与排期方案（用于后续调整）

| 方案 | 特点 | 适用场景 | 风险 |
|---|---|---|---|
| A 稳健推进（推荐） | P0 优先、分阶段发布、质量门禁严格 | 当前阶段（既要速度又要稳定） | 发布节奏略慢于激进方案 |
| B 激进并行 | 多模块并行、压缩周期 | 资源充足、容错高 | 返工与质量风险高 |
| C 保守防御 | 先补稳定性再做新功能 | 线上故障高发时 | 产品创新节奏偏慢 |

推荐结论：采用 A 稳健推进，以 Phase 5 为关键拐点，先保证 A2A/记忆/自主引擎与安全基线，再放大协作和云原生收益。
