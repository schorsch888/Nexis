## 2. 发现的问题（按严重程度）

## P0（必须立即处理）

1. `multi-tenant + persistence-sqlx` 组合特性不可编译（Feature Flag 断裂）
- 证据：`cargo check -p nexis-gateway --features "multi-tenant,persistence-sqlx"` 报错。
- 关键位置：`crates/nexis-gateway/src/db/mod.rs:255`、`crates/nexis-gateway/src/db/mod.rs:324`、`crates/nexis-gateway/src/db/mod.rs:406`
- 问题表现：
  - `RoomRepository/MessageRepository/MemberRepository` 在 `multi-tenant` 下要求实现 tenant 方法，但 SQLx 实现缺失。
  - 多个结构体初始化缺 `tenant_id` 字段。
  - `Uuid` 在 SQLx 路径未导入（`db/mod.rs:257` 等报错）。
- 风险：生产特性组合不可用，发布策略不可预测，测试覆盖失真。

2. 认证链路未接入路由，实际 API 无有效鉴权
- 证据：`AuthenticatedUser` 提取器直接返回 `401` 占位实现。
- 关键位置：`crates/nexis-gateway/src/auth/mod.rs:145`
- 佐证：`router` 未使用 `AuthenticatedUser`/`TenantExtractor` 作为 handler 参数：`crates/nexis-gateway/src/router/mod.rs:235`、`280`、`329` 等。
- 风险：文档宣称“所有 API 需要 JWT”，但代码未落地，属于安全承诺与实现不一致。

3. 协议规范与实现严重漂移（同名 NIP，不同语义）
- `MemberType` 不一致：
  - 规范：`human | ai | agent | system`（`protocol/nexis-id.md:15`）
  - 实现：`Human | Agent | Bot`（`crates/nexis-protocol/src/lib.rs:14`）
- `MessageContent` 不一致：
  - 规范包含 markdown/data/media/tool_call/tool_result/system/thinking/stream_*（`protocol/nexis-msg.md`）
  - 实现仅 `Text/Code/Tool`（`crates/nexis-protocol/src/lib.rs:147`）
- 风险：客户端/服务端互操作失败，后续版本升级成本急剧上升。

## P1（高优先）

1. 协议版本兼容策略缺失
- 关键位置：`crates/nexis-protocol/src/lib.rs`（整体）
- 问题：无 `protocol_version` / `schema_version` / 能力协商字段；无“向后兼容规则”代码化约束。
- 对标差距：
  - Matrix 有事件 schema + room version 演进机制。
  - Discord Gateway 有版本化接口与 opcode/intent 协议边界。
  - Slack Events 有 envelope + event type 兼容策略。

2. 错误模型不统一，协议层返回裸字符串
- 关键位置：`crates/nexis-protocol/src/lib.rs:198`
- 问题：`Message::validate() -> Result<(), String>`，错误不可结构化、不可国际化、不可机器路由。
- 风险：跨服务错误处理不可观测，难以建立稳定 SDK。

3. 多租户隔离只在“部分 in-memory 测试路径”成立，SQL 侧未形成强制约束
- 关键位置：
  - 迁移已添加 `tenant_id`：`crates/nexis-gateway/migrations/0002_add_tenant_id.sql`
  - SQLx 查询仍大量不带 tenant 条件：`crates/nexis-gateway/src/db/mod.rs:276`、`353`、`427` 等。
- 风险：一旦接入真实持久化，极易发生跨租户读取。

4. 入口配置默认过宽
- 关键位置：`crates/nexis-gateway/src/main.rs:29`
- 问题：`CorsLayer::new().allow_origin(Any).allow_methods(Any)`。
- 风险：浏览器侧跨域攻击面扩大，不符合默认安全基线。

## P2（中优先）

1. 协议解析存在“静默降级”
- 关键位置：`crates/nexis-protocol/src/lib.rs:57-64`
- 问题：`member_type()` 遇到未知值默认返回 `Human`。
- 风险：身份类型误判（权限/审计污染）。

2. API 文档与实现不一致
- 文档：`docs/en/api/reference.md` 声称 `GET /rooms`、`DELETE /rooms/{id}` 等。
- 实现：`router` 当前仅 `POST /v1/rooms`、`POST /v1/messages`、`GET /v1/rooms/:id` 等（`crates/nexis-gateway/src/router/mod.rs:200-207`）。
- 风险：外部集成失败，客户端重试行为不可控。

3. 测试组织存在“孤儿测试”迹象
- `tests/security/tenant_isolation_test.rs` 与当前 `TenantId` API 语义不匹配（如 `TenantId::new(Uuid::new_v4())`，实际是 `TenantId::new()` / `from_uuid`）。
- workspace 级别难以直接作为独立 test target 执行。
- 风险：安全回归测试可能未进入标准流水线。

## P3（建议优化）

1. 模块存在“已声明未集成”
- `server` 模块基本为空：`crates/nexis-gateway/src/server/mod.rs:1`
- `connection` 模块标注 pending integration：`crates/nexis-gateway/src/connection/mod.rs:6`

2. 上下文管理策略未闭环
- `Summarize` 分支仍 TODO：`crates/nexis-context/src/manager.rs:64`

## 4. 协议改进建议（详细设计，对标 Matrix/Discord/Slack）

## 4.1 目标能力

参考三类成熟系统，Nexis 协议层应满足：
- 可演进 envelope（不破坏旧客户端）
- 强类型错误与可观测性字段
- 扩展点 namespace 化
- 事件/消息解耦（消息只是事件的一种）

## 4.2 建议的 NIP v2 协议骨架

1. 统一事件 Envelope（类似 Matrix/Slack envelope）

```json
{
  "eventId": "evt_01J...",
  "eventType": "message.created",
  "protocolVersion": "2.0",
  "schemaRevision": 1,
  "tenantId": "t_123",
  "roomId": "room_abc",
  "sender": "nexis:ai:openai/gpt-4.1",
  "occurredAt": "2026-02-23T08:00:00Z",
  "traceId": "trc_...",
  "content": { "type": "text", "text": "hello" },
  "relations": {
    "replyTo": "evt_prev",
    "threadId": "thr_1"
  },
  "extensions": {
    "nexis.ai.usage": { "inputTokens": 12, "outputTokens": 30 }
  }
}
```

2. 内容类型分层
- Core（协议必须支持）：`text`, `code`, `tool_call`, `tool_result`, `system`
- Optional（能力协商）：`markdown`, `media`, `thinking`, `stream_chunk`
- 规则：未知 `content.type` 必须可忽略且保留原始 payload（forward compatible）

3. 错误模型标准化

```json
{
  "error": {
    "code": "NEXIS.PROTOCOL.INVALID_FIELD",
    "message": "roomId is required",
    "field": "roomId",
    "retryable": false,
    "details": { "expected": "non-empty string" }
  }
}
```

错误码建议分层：
- `NEXIS.AUTH.*`
- `NEXIS.PERMISSION.*`
- `NEXIS.TENANT.*`
- `NEXIS.PROTOCOL.*`
- `NEXIS.RUNTIME.*`

4. 版本兼容策略（必须写入 NIP）
- Minor 版本：只能新增可选字段/枚举值，不得修改语义。
- Major 版本：允许破坏性变更，需提供迁移窗口与双栈解析器。
- 服务端规则：
  - 接收端必须忽略未知字段。
  - 发送端不得发送超出对端能力协商的必需字段。

5. 能力协商（借鉴 Discord Gateway intents 思路）
- `HELLO` / `CAPS` 交换：声明支持的 `eventType`、`content.type`、`compression`、`streaming`。
- 降级策略：不支持时回落到 `text` 或非流式模式。

## 4.3 协议层代码拆分建议

当前 `crates/nexis-protocol/src/lib.rs` 单文件承载过多职责，建议拆分：
- `member.rs`：`MemberId/MemberType`
- `message.rs`：`MessageEnvelope/MessageContent`
- `permission.rs`：`Permissions/Action`
- `error.rs`：`ProtocolError`
- `version.rs`：版本与兼容策略

## 5. 实施计划（优先级排序）

## Phase 0（1-2 天，阻断风险）

1. 修复 `multi-tenant + persistence-sqlx` 编译失败。
2. 将鉴权提取器接入所有写接口。
3. 收紧 CORS 默认策略。

验收标准：
- `cargo check -p nexis-gateway --features "multi-tenant,persistence-sqlx"` 通过。
- 未携带 JWT 的写请求返回 401。

## Phase 1（3-5 天，协议对齐）

1. 冻结 NIP 词汇表（member type/content type/error code）。
2. 发布 `nexis-protocol` v2 草案（双栈兼容）。
3. 建立协议一致性测试（文档示例 JSON 必须能被解析）。

验收标准：
- `protocol/*.md` 示例全部有自动化 round-trip 测试。
- 旧 payload 在兼容模式可解析。

## Phase 2（1-2 周，架构升级）

1. 路由 DTO 向 `nexis-protocol` 类型收敛。
2. tenant 隔离在 SQL 层强制执行（查询必带 tenant 过滤）。
3. 新增 feature matrix CI。

验收标准：
- 四组 feature 组合都跑通编译 + 核心测试。
- 跨租户访问有集成测试覆盖（真实数据库）。

## Phase 3（持续演进）

1. 引入协议能力协商。
2. 引入审计事件标准字段（traceId/actor/tenant/action/result）。
3. 完成 context summarize 能力闭环。

## 6. 与大厂协议设计对标结论

- 与 Matrix 的差距：缺少“事件版本化 + 长期兼容机制”。
- 与 Discord 的差距：缺少“连接层能力协商 + 明确网关事件语义”。
- 与 Slack 的差距：缺少“稳定事件 envelope 与可扩展事件类型治理”。

Nexis 当前最关键的改造方向不是“加更多字段”，而是先建立：
- 统一协议词汇表
- 严格版本策略
- 结构化错误模型
- 安全与多租户在运行时的强制执行

以上四项完成后，协议层才能进入可持续演进轨道。
